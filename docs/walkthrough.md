# Denko1-Tracker 開発完了報告

第一種電気工事士 試験対策トラッカー「Denko1-Tracker」の開発が、Phase 6 まで完全に完了しました。

## 🚀 成果物の概要

本アプリケーションは、第一種電気工事士の合格を目指す学習者が、自身の「弱点」を視覚的に把握し、効率的に学習を進めるためのツールです。

### 主な機能
1.  **筆記試験管理**: 過去問の得点を分野別（計算・機器・施工図・法令）に記録。実際の試験配点に基づいた重み付け計算に対応。
2.  **弱点分析エンジン**: 最新の記録から正答率を算出し、レーダーチャートで視覚化。
3.  **最優先復習項目の自動特定**: 最も正答率が低い分野をアラート表示。
4.  **技能試験管理**: 公表問題の施工タイムを記録し、制限時間内での合格をサポート。
5.  **完全なCRUD操作**: 登録・閲覧・編集・削除すべてに対応した本格的なデータ管理。
6.  **セキュアなマルチユーザー対応**: 各ユーザーが個別に学習記録を蓄積・保護。他ユーザーのデータは閲覧・操作不可。
7.  **CSVデータ出力**: 記録した学習データを一括ダウンロードし、Excelでの詳細分析が可能。
8.  **全画面モバイル最適化**: ログイン画面からダッシュボードまで、全ページをスマートフォン対応。
9.  **試験カウントダウン**: 学科・技能試験の目標日を設定し、ダッシュボードで残り日数を可視化。

## 📝 運用・メンテナンス履歴

### 2026年4月10日：データベースの永続化と引っ越し（Supabaseへの移行）
*   **課題**: Render Managed PostgreSQL の無料枠に「作成から30日間（5月7日まで）でデータ削除」という制限があることが判明。
*   **リスク**: 5月7日は学科試験の当日であり、試験直前にデータが喪失するリスクがあった。
*   **対策**: 永続的な無料枠を提供する **Supabase (PostgreSQL)** への移行を緊急実施。
*   **実施内容**:
    -   Javaによる一時的なデータエクスポートツールを作成し、SQLダンプを生成。
    -   Supabase側でスキーマを再現し、インポートを完了。
    -   `application-prod.properties` を修正し、環境変数による動的な接続設定（SSL対応）を導入。
*   **結果**: データの永続性が確保され、試験当日およびその後の技能試験期間も安心して利用できる環境が整った。

### 2026年4月10日：移行後のログイン不具合解消（認証エンジンの安定化）
*   **課題**: Supabase への移行直後、全ユーザーのログインおよび新規登録が「パスワード不一致」で失敗する現象が発生。
*   **原因分析**: 
    - Supabase の Row Level Security (RLS) により、アプリからの読み取りが制限されていた。
    - 明示的な認証プロバイダー構成が Spring Security の自動構成と干渉していた可能性。
*   **解決策**:
    - **RLS の無効化**: `ALTER TABLE DISABLE ROW LEVEL SECURITY` を実行。
    - **SecurityConfig の簡略化**: 明示的な `DaoAuthenticationProvider` の定義を廃止し、Spring Boot の標準的な自動構成に委ねる形に変更。
    - **スキーマ同期の強制**: `ddl-auto=update` を一時的に有効化し、Javaエンティティと実テーブル間の微細な不整合を自動修復。
*   **結果**: ログインおよび新規登録機能が完全に復旧し、全レコードが正しく参照できることを確認。

### 2026年4月11日：Gitリポジトリの整理とインフラ構成の最適化
*   **課題**: 
    - 開発が Phase 7 まで進んでいたが、フェーズごとのブランチが作成されておらず、履歴の管理が不透明だった。
    - `render.yaml` が旧構成（Render Managed DB）のまま残っており、インフラ構成と実態が乖離していた。
    - プロジェクトに関係のない Python 関連のファイルが混入していた。
*   **対策**:

    - **不要ファイルのクリーンアップ**: `hello.py`、`python-practice/`、`venv/` などを一括削除。
    - **`render.yaml` の最適化**: 旧データベース定義を削除し、Supabase 向けの環境変数構成へ刷新。機密情報は手動入力を維持する安全な設定に変更。
*   **結果**: 
    - リポジトリがクリーンになり、開発フェーズごとの振り返りが可能になった。
    - インフラ構成が「コード」として正しく定義され、保守性が向上した。

### 2026年4月13日：レーダーチャートの表示不具合修正と UX 改善
*   **課題**: レーダーチャートで 80% と表示されるべき項目が 0% になるなど、データの不整合や見づらさが報告された。
*   **原因分析**: 
    - データの取得順序が「年度・回数順」となっており、直近の更新内容が反映されない場合があった。
    - 項目名と数値のリストを別々に渡していたため、シリアライズ時に順位がずれるリスクがあった。
    - チャートの目盛り（ticks）が非表示で、正確な値が読み取りにくかった。
*   **対策**:
    - **更新日時フィールドの導入**: 全レコードに `updatedAt` を追加し、最新の更新データをグラフに採用。
    - **データ転送の安定化**: 明示的に `ArrayList` を用いて、データの順列をバックエンド・フロントエンド間で完全に固定。
    - **チャート表示の調整**: Chart.js の設定を最適化し、目盛りの表示と `0-100` スケールの固定を実施。
- **追加対策（緊急）**: デプロイ後の 500 エラーに対し、`application-prod.properties` の `ddl-auto` を `update` に変更することでスキーマ不整合を解消。
*   **結果**: データの整合性が100%保証され、かつ本番環境での稼働が完全に復旧した。

### 2026年6月5日：筆記試験履歴のソート順変更と分析元データの表示改善
*   **課題**:
    - 筆記試験履歴テーブルで、同じ過去問を何度も解いた際の進捗（回数を追うごとの点数の推移）を時系列で把握しづらかった。
    - レーダーチャートでどの過去問データ（年度・期・回数）が分析対象になっているのかが、視覚的に分かりにくかった。
*   **対策**:
    - **ソートロジックの改善**: `WebController` のデータ取得処理にカスタム `Comparator` を導入。表示順およびCSVエクスポート順を「1. 年度（降順：新しい順）」「2. 期（昇順：上期→下期→午前→午後）」「3. 挑戦回数（昇順）」で並べ替えるように修正。
    - **分析元データの表示**: レーダーチャートの直下に、現在グラフに採用されているレコードの年度・期・回数情報（例: `2025年度 上期 1回目`）を太字で表示。
    - **テストコードの強化**: ソートロジックの正確性を検証する `WebControllerTest` を新規作成。また、不整合が起きていた既存の `WeaknessAnalysisServiceTest` を最新の配点ロジックに適合するよう修正。
*   **結果**:
    - 同じ年度・期の過去問において、回数を重ねるごとの得点の推移がテーブル上で一目で確認できるようになり、実力向上の度合いが明確になった。
    - チャートが示す「現在の弱点」がどの時点のデータに基づくものかが明示され、学習計画がより立てやすくなった。

### 2026年6月25日：技能試験の制限時間超過時の「時間切れ」表示対応
*   **課題**: 技能試験の履歴で、完成タイムが制限時間（60分）を超過している場合でも「合格」や「欠陥あり」とだけ表示され、時間制限を守れなかったことが判別しづらかった。
*   **対策**:
    - `index.html` にて、技能試験レコードの完成タイムが60分を超える（`completionTimeMinutes > 60`）場合の条件分岐を追加。
    - 条件に当てはまる場合は、合否ステータスに関わらず結果表示を「時間切れ」とし、赤字の警告色（`class="fail"`）で強調するよう修正。
*   **結果**: 制限時間内に完成できたかどうかが一目で分かるようになり、本番試験の合格基準に則したより実用的な進捗管理が可能となった。

### 2026年7月5日：タイムゾーン不整合によるカウントダウン表示のバグ修正
*   **課題**: サーバーとユーザー（日本時間）間のタイムゾーンの不一致（時差9時間）により、日本時間の午前9時前にアクセスした際、試験日翌日になっても「本日試験日！」と表示され、さらに翌日に「試験終了」と表示されるなど、カウントダウンの進行が1日遅れる現象が発生していた。
*   **対策**:
    - `Denko1TrackerApplication.java` において、アプリケーション起動時に JVM のデフォルトタイムゾーンを `Asia/Tokyo`（JST）に設定。
    - `WebController.java` におけるカウントダウン計算で、明示的に `LocalDate.now(ZoneId.of("Asia/Tokyo"))` を呼び出すことで、実行環境（UTC等のサーバー）に関わらず確実に日本時間の現在日を基準に計算するように修正。
    - `DataInitializer.java` のテストデータ作成でも `ZoneId` を明示指定した `LocalDate` および `LocalDateTime` の現在日時取得を行うように変更。
*   **結果**: サーバーが UTC で稼働している状態でも、カウントダウン（「あと何日」「本日試験日！」「試験終了」）の切り替わりが日本時間の午前0時基準で正しく行われるようになり、正確な表示が保証された。
